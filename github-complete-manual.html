What is GitHub?

Understanding GitHub from zero - no experience needed!

The Simple Explanation

GitHub is a website where people store their code and files, keep track of every change they make, and work together with other people on projects. It is like Google Drive, but supercharged with special powers for tracking changes.

Kid-Friendly Analogy

Imagine you are writing a story in a notebook. Every time you change something, you take a photo of that page. Now you have photos of every version of your story. If you mess up, you can go back to any photo! GitHub does exactly this, but for code and files. And the best part? Your friends can also write in their own copies and then you combine the best parts together!

The Three Big Ideas of GitHub

  • Version Control - GitHub remembers EVERY change you ever make. You can go back in time to any previous version. It is like having an undo button that goes back forever!
  • Collaboration - Multiple people can work on the same project at the same time without stepping on each other's toes. Everyone gets their own workspace.
  • Sharing - You can show your projects to the world, or keep them private. Open source means anyone can see and learn from your code.

GitHub vs Git - What is the Difference?

Git is the tool (the engine) that tracks changes to your files on your computer. GitHub is the website (the garage) where you park and share your Git projects online. Think of Git as the car engine, and GitHub as the highway where all the cars drive and meet each other.

Pro Tip

You do NOT need to know how to code to use GitHub! Many writers, designers, and teachers use GitHub to track documents, share resources, and collaborate on non-code projects.

Repositories (Repos)

Your project's home - where all your files live together

What is a Repository?

A repository (or "repo" for short) is like a special folder that holds ALL the files for one project. But it is way more powerful than a regular folder because it also remembers every change ever made to those files.

Kid-Friendly Analogy

Think of a repository like a magical backpack. You put all your school supplies for ONE subject inside it. The backpack remembers every time you added or removed something, and you can magically go back to how it looked on any day!

Types of Repositories

TypeWho Can See ItBest For
PublicEveryone on the internetOpen source projects, portfolios, learning resources
PrivateOnly you and people you invitePersonal projects, company code, secret work

What is Inside a Repository?

  • Files and Folders - Your actual code, images, documents, anything!
  • README.md - A welcome page that explains your project (like a book cover)
  • Commit History - A timeline of every change ever made
  • Branches - Different versions of your project running side by side
  • Issues - A to-do list or bug tracker for your project
  • Settings - Controls for who can do what in your repo

How to Create a Repository

Go to GitHub.com

Log in to your GitHub account. Click the green "New" button or the "+" icon in the top right corner.

Name Your Repository

Give it a short, clear name. Use hyphens instead of spaces (like "my-cool-project"). Add a description so people know what it is about.

Choose Public or Private

Public means anyone can see it. Private means only you and invited people. You can change this later!

Add a README

Check the box that says "Add a README file". This creates a welcome page for your project automatically.

Click Create Repository

Hit the green button and your repo is born! You now have a home for your project.

Pro Tip

Always add a README file! It is the first thing people see when they visit your repo. Write what your project does, how to use it, and how to contribute.

Commits

Saving your work with a message - like taking snapshots in time

What is a Commit?

A commit is when you save changes to your project AND write a little note about what you changed. It is the most important action in GitHub. Every commit creates a permanent record in your project's history.

Kid-Friendly Analogy

Imagine you are building with LEGO blocks. Every time you add or change some blocks, you take a photo and write on the back what you did: "Added a red tower" or "Fixed the broken bridge". Now you have a photo album of your entire building process! A commit is exactly like that photo with a note.

What Makes a Good Commit?

  • Small and focused - Change one thing at a time, not everything at once
  • Clear message - Write what you did so future-you understands it
  • Meaningful - Each commit should have a purpose

Good vs Bad Commit Messages

Bad MessageGood Message
"fixed stuff""Fix login button not responding on mobile"
"update""Add search feature to navigation bar"
"asdfgh""Remove unused CSS styles from homepage"
"changes""Update pricing table with new monthly plans"

How to Make a Commit on GitHub.com

Edit a File

Open any file in your repo and click the pencil icon to edit it.

Make Your Changes

Type your changes in the editor. You will see the changes highlighted.

Write a Commit Message

Scroll down to the "Commit changes" section. Write a short, clear message about what you changed.

Click Commit Changes

Click the green button. Your changes are now saved forever in the project history!

Important

You cannot undo a commit, but you CAN revert it (create a new commit that undoes the old one). Think before you commit!

Branches

Work on new ideas without breaking your main project

What is a Branch?

A branch is a separate copy of your project where you can make changes without affecting the main version. When you are happy with your changes, you merge the branch back into the main project.

Kid-Friendly Analogy

Imagine you have a coloring book. The "main" branch is your original coloring book. When you want to try coloring a page with different colors, you photocopy that page (create a branch). You color the copy however you want. If it looks amazing, you replace the original page with your new version (merge). If it looks terrible, you just throw away the copy and your original is still perfect!

The Main Branch

Every repo starts with one branch called main (sometimes called "master" in older repos). This is your official, stable version. You should never experiment directly on main - always create a new branch first!

Common Branch Naming

Branch NamePurpose
mainThe stable, production-ready version
feature/add-loginAdding a new feature (login page)
fix/broken-buttonFixing a bug (broken button)
devDevelopment branch for testing new things

How to Create a Branch

Go to Your Repository

Navigate to your repo on GitHub.com

Click the Branch Dropdown

Click where it says "main" near the top left of your file list

Type a New Branch Name

Type a descriptive name like "feature/new-homepage"

Click Create Branch

Click the option to create the new branch. You are now working on your new branch!

Pro Tip

Always create a new branch before making changes. This keeps your main branch clean and stable. If something goes wrong on your branch, main is still perfectly fine!

Pull Requests (PRs)

The polite way to suggest changes to a project

What is a Pull Request?

A Pull Request (or PR) is how you tell other people: "Hey, I made some changes on my branch and I think they are good enough to be added to the main project. Can you please review them?" It is the bridge between your branch and the main branch.

Kid-Friendly Analogy

Imagine you are in a group art project. You drew something cool on your piece of paper (your branch). Now you raise your hand and say: "Teacher, I would like to add my drawing to the big poster (main branch). Can everyone look at it first?" Your classmates look at it, maybe suggest small improvements, and when everyone agrees it is good, the teacher tapes it onto the poster. That is a Pull Request!

The Pull Request Workflow

Create a Branch and Make Changes

Work on your feature or fix in a separate branch

Open a Pull Request

Go to your repo, click "Pull requests" tab, then "New pull request". Select your branch.

Describe Your Changes

Write a title and description explaining what you changed and why

Request Reviewers

Ask teammates to review your code by adding them as reviewers

Discuss and Improve

Reviewers leave comments. You can make more changes based on feedback.

Merge the Pull Request

Once approved, click the green "Merge" button to add your changes to main!

PR Status Labels

StatusMeaning
OpenThe PR is waiting to be reviewed and merged
MergedThe changes have been accepted and added to main
ClosedThe PR was rejected or cancelled without merging

Issues

Your project's to-do list and bug tracker

What are Issues?

Issues are like sticky notes for your project. They can be bug reports ("This button is broken!"), feature requests ("It would be cool if we added dark mode"), questions, or tasks to complete.

Kid-Friendly Analogy

Think of Issues like a classroom suggestion box and to-do list combined. Anyone can write a note saying "The clock is broken" (bug report) or "Can we get a fish tank?" (feature request). The teacher reads them, assigns people to handle them, and checks them off when done.

Parts of an Issue

  • Title - A short summary of the problem or idea
  • Description - Detailed explanation with steps to reproduce bugs
  • Labels - Color tags like "bug", "enhancement", "help wanted"
  • Assignees - Who is responsible for handling this issue
  • Milestones - Which project goal this issue belongs to
  • Comments - Discussion thread where people talk about the issue

Common Labels

LabelColorMeaning
bugRedSomething is broken and needs fixing
enhancementBlueA new feature or improvement idea
help wantedGreenThe owner needs help with this
good first issuePurpleEasy task perfect for beginners
documentationYellowNeeds better docs or explanations
Pro Tip

When reporting a bug, always include: what you expected to happen, what actually happened, and the steps to reproduce the problem. Screenshots help a lot!

GitHub Pages

Turn your code into a live website for free!

What is GitHub Pages?

GitHub Pages is a FREE service that takes your HTML, CSS, and JavaScript files from a repository and publishes them as a real website that anyone in the world can visit. No need to buy hosting or a domain name!

Kid-Friendly Analogy

Imagine you drew a poster for a school event. GitHub Pages is like the school hallway bulletin board - you pin your poster there and now EVERYONE walking by can see it. Your code files are the poster, and GitHub Pages is the bulletin board that shows them to the world!

Your Website URL

When you enable GitHub Pages, your website gets a free URL that looks like this:
https://yourusername.github.io/your-repo-name/
For example: https://nicoschotje.github.io/nico-digital-products/

How to Enable GitHub Pages

Go to Repository Settings

Click the "Settings" tab in your repository

Click on Pages

In the left sidebar under "Code and automation", click "Pages"

Select Source Branch

Under "Branch", select "main" from the dropdown and keep "/ (root)" for the folder

Click Save

Hit Save and wait a few minutes. Your site will be live!

Visit Your Site

Click "Visit site" or go to https://yourusername.github.io/repo-name/

Pro Tip

To access a specific HTML file, add the filename to the URL. Example: https://yourusername.github.io/repo-name/my-page.html

GitHub Actions

Automate tasks like a robot assistant for your project

What are GitHub Actions?

GitHub Actions are automatic tasks that run whenever something happens in your repository. They can test your code, deploy your website, send notifications, and much more - all automatically!

Kid-Friendly Analogy

Imagine you have a robot helper in your room. You tell it: "Every time I put my homework on the desk (push code), check it for mistakes (run tests) and then put it in my backpack (deploy)." The robot does this automatically every single time. That is what GitHub Actions does for your code!

Common Uses

  • Testing - Automatically check if your code works correctly
  • Building - Compile and package your app automatically
  • Deploying - Push your website live whenever you update code
  • Linting - Check code style and formatting rules
  • Notifications - Send alerts when something happens

How Actions Work

ConceptWhat It Means
WorkflowA set of instructions saved as a YAML file in your repo
TriggerThe event that starts the workflow (like a push or PR)
JobA group of steps that run together
StepA single task within a job
RunnerThe computer that runs your workflow
Note

GitHub gives you 2,000 free minutes per month for Actions on free accounts. That is plenty for most personal projects!

Forks & Stars

The social features of GitHub

Stars - Bookmarking Projects You Like

Starring a repo is like giving it a thumbs up or bookmarking it. It helps you find it later and shows the creator that you appreciate their work. The more stars a project has, the more popular it is!

Forks - Making Your Own Copy

Forking a repo creates a complete copy of someone else's project under YOUR account. You can then change it however you want without affecting the original. It is the foundation of open source collaboration!

Kid-Friendly Analogy

Stars are like putting a gold sticker on your favorite book in the library. Forks are like photocopying the entire book so you can write notes in your own copy, highlight things, or even add new chapters!

When to Fork

  • You want to contribute to someone else's project
  • You want to use someone's project as a starting point for your own
  • You want to experiment with someone's code without affecting the original

Fork + PR = Open Source Magic

Fork the Repository

Click the "Fork" button on any public repo to get your own copy

Make Changes in Your Fork

Edit files in your forked copy however you want

Submit a Pull Request

Send your changes back to the original project for review

GitHub Projects

Organize your work like a pro with project boards

What are GitHub Projects?

GitHub Projects is a built-in project management tool. It gives you a board (like a digital whiteboard) with columns where you can track tasks, organize work, and see progress at a glance.

Kid-Friendly Analogy

Think of it like a big whiteboard in your room with three columns: "To Do", "Doing", and "Done". You write tasks on sticky notes and move them from left to right as you work. GitHub Projects is the digital version of this!

Default Columns

ColumnPurpose
To DoTasks that need to be done but have not started yet
In ProgressTasks currently being worked on
DoneCompleted tasks
Pro Tip

You can link Issues and Pull Requests directly to your project board. When a PR is merged, the linked card automatically moves to "Done"!

Settings & Security

Protect your account and configure your repositories

Account Security

  • Two-Factor Authentication (2FA) - Adds an extra layer of protection. Even if someone knows your password, they cannot log in without your phone code. ALWAYS enable this!
  • Strong Password - Use a unique, long password for GitHub
  • SSH Keys - A secure way to connect your computer to GitHub without typing your password every time
  • Personal Access Tokens - Special passwords for apps that need to access your GitHub

Repository Settings

Key Settings You Should Know

SettingWhat It DoesRecommendation
VisibilityPublic or Private repoPrivate for personal, Public for open source
Branch ProtectionPrevents direct pushes to mainEnable for team projects
GitHub PagesPublish repo as a websiteEnable for HTML projects

Security Warning

NEVER commit passwords, API keys, or secrets to your repository! Use environment variables or GitHub Secrets instead. Once a secret is in your commit history, it is very hard to remove completely.

Best Practices

The golden rules for using GitHub like a professional

The 10 Golden Rules of GitHub

Always Use Branches

Never work directly on main. Create a branch for every feature, fix, or experiment.

Write Clear Commit Messages

Start with a verb: "Add", "Fix", "Update", "Remove". Make it clear what changed and why.

Commit Often, Push Often

Small, frequent commits are better than one massive commit. This makes it easier to find bugs.

Always Add a README

Every repo needs a README that explains what the project does, how to set it up, and how to use it.

Use .gitignore

Create a .gitignore file to exclude files that should not be tracked (like passwords, build files, node_modules).

Review Before Merging

Always have at least one other person review your Pull Request before merging.

Use Issues for Everything

Track bugs, features, and tasks with Issues. It keeps everything organized and visible.

Protect Your Main Branch

Set up branch protection rules so nobody can push directly to main without a review.

Never Commit Secrets

API keys, passwords, and tokens should NEVER be in your code. Use environment variables instead.

Enable 2FA

Two-Factor Authentication protects your account. Enable it immediately!
